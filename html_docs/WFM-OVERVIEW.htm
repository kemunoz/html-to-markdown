Work Flow Managment Overview

Data Entry Procedures | Reports and Inquiries | Update Processes

Purpose
The Work Flow Management module provides the infrastructure required to provide real time feedback and notification based on user defined events. Events may be defined to notify one or more users of specific events that have occurred. For example, an event can be setup to notify a specific user anytime an inventory item falls below its safety stock level. Events may be defined as persistent so that they remain in effect until specifically removed or setup as one time events.

Alerts are defined using the Process Alert Definition procedure (PADEF.E). The definitions, or templates, are then used by individual users to indicate that an alert is to be made active using the Process Alert procedure (PALERT.E). The user answers questions that were defined in the template and adjusts the text of the message to be sent when the event occurs. These specific instructions are then used to monitor activity on the system until the event occurs at which time a message is sent to all of the users specified on the distribution list. If the message was defined as persistent it will continue monitoring the system for another occurrence of the event otherwise it is deleted.

In addition to notifying the user of an event the message sent to the user may also define a link to a procedure and a data entry script. For example, if a message indicates that an inventory level has gone below its minimum you could define a link to the inventory inquiry procedure and pass it the part number in question allowing the user to activate the link to view the inventory status.

The primary user interface to the module is the Task Manager (TM.E) which includes an email like interface for receiving and sending messages within the system. Messages to a specific user may also be directed to a regular email account. The Task Manager also includes a task list and calendar. While the Task Manager is running it will check for new messages on a user defined time period and notify the user that a new message has been received by placing a yellow envelope icon on the lower right side of the task bar in each of the open application windows.

The Task Manager (TM.E) is not meant to be a replacement for traditional email and does not process email messages sent from outside the system, but it can provide a means of internal communication for personnel that would not normally have regular email and has the added benefit of being free from spam and other parasitic nuisances that interfere with standard email.

Prerequisites
None.

Data Files
There are ten primary data files which maintain the data directly associated with the module.

The TM file maintains each users task list and is accessed using the TM.E procedure.

The CALENDAR file maintains records related to each users calendar and is accessed using the TM.E procedure.

The MESSAGE file maintains a record for each users messages and is accessed using the TM.E procedure.

The PADEF file contains the alert templates defined using PADEF.E and used by PALERT.E.

The PALERT file contains all of the alerts that are being monitored for an event that matches their criteria and are created using the PALERT.E procedure.

The MSGQUE file is a holding file for messages that have not been delivered to the user.

The PAQUE file is a holding file for activity that is being monitored and is awaiting processing by a phantom process to determin if it meets the criteria of any of the alerts defined in the PALERT file.

The PARECQUE file is related to the PAQUE file and contains the before and after image of the data record associated with the event identified in the PAQUE record.

The PATEMPQUE file performs the same function as the PARECQUE file except it contains the data from the temporary fields of the procedure that triggered the event.

The MSGTEMP file is used for managing messages during their creation through the MESSAGE.E procedure.

System Interface
The Work Flow Management module interfaces to all other system modules based on the notification events defined.

User Interface
Administrative users will utilize the Process Alert Definition procedure (PADEF.E) to define templates for the alerts that may be generated. Administrators and, depending on your organizational structure, certain end users will use the Process Alert Definition procedure to setup specific events to be monitored. End users wil use the Task Manager (TM.E) and or regular email to receive notification of events.

Setup Procedures
The MSG.CONTROL procedure must be setup and started so that messages generated by events or other users will be delivered.

The PALERT.CONTROL procedure must be setup and started so that alerts that have been defined are monitored and acted upon.

If the ability to forward messages to external email is required the MAIL.CONTROL procedure must be setup.

Note: The phantom processes launched by MSG.CONTROL and PALERT.CONTROL may also be setup to automatically start by including the proper parameters with the ACE$COLDSTART program in the database startup process.

Flow of Activities
Aside from initially setting up the work flow parameters the flow of activities for this module is dictated by the activities triggered by the alert messages generated and are unique to each enterprise.
